# Plugin Builder — Design Spec

## Обзор

Плагин `plugin-builder` — централизованный инструмент для жизненного цикла плагинов в маркетплейсе `anton-toolkit-marketplace`. Закрывает три задачи: создание нового плагина с нуля по task-centric интервью, проактивное расширение существующего плагина, реактивное исправление плагина на основе инцидента в текущей сессии.

Главный мотив создания — устранить дублирование: скилл `improve-plugin` сейчас скопирован в каждом плагине с захардкоженным путём, что не масштабируется. `plugin-builder` выносит эту логику в одно место и добавляет отсутствующие на сегодня возможности создания и расширения плагинов.

## Ключевые инварианты

- **Язык инструкций:** SKILL.md, агенты, references — всегда **английский** (экономия токенов, лучшее следование правилам). Диалог с пользователем в чате, коммит-сообщения, русские триггер-фразы в описаниях — **русский**.
- **Модель:** `improve-plugin` — обязательно **Opus** (точность формулировок критична, Sonnet на этой задаче наблюдаемо ошибается). `create-plugin` и `extend-plugin` — default.
- **Диалоговые компоненты — скиллы** (`create-plugin`, `extend-plugin`). Автономные — агенты (`improve-plugin`). Правило: многоходовой диалог не живёт в агенте, потому что агент теряет контекст между вызовами.
- **Git-флоу автоматический:** pull → правка → валидация → commit → push. Никакого ручного diff-ревью (пользователь не читает английский). Вместо этого — усиленная pre-flight валидация и русское саммари с hash'ем коммита для возможного отката.
- **Spec-файлы для генерируемых плагинов не создаются.** Фаза ревью — русский конспект в чате на подтверждение.

## Архитектура компонентов

### `create-plugin` — скилл (default model)

**Триггер:** свежая сессия, плагина ещё нет. Фразы вида «создай плагин», «нужен новый скилл для…», «хочу плагин».

**Флоу интервью (6 фаз):**

1. Открытый вопрос на русском: «Расскажи, что за плагин и какую задачу он должен решать».
2. Парсинг ответа под капотом по чек-листу из 4 обязательных полей.
3. Активное уточнение — по одному вопросу за сообщение, только на непокрытые/частично покрытые поля. HTML-комментарий `<!-- uncovered: [list] -->` в конце каждого сообщения модели для самоподсчёта (пользователь не видит).
4. Русский конспект → подтверждение/корректировки.
5. Генерация файлов плагина: `plugin.json` (версия 0.1.0), SKILL.md / agent.md компонентов (на английском), обновление `marketplace.json`.
6. Pre-flight валидация → git pull → commit → push → русское саммари с hash'ем.

**Чек-лист обязательных полей (task-centric):**

1. **Задача** — какую проблему плагин решает (не «что делает», а «что мешает без него»).
2. **Когда запускать** — контекст ситуаций срабатывания (из него скилл сам выводит конкретные триггер-фразы).
3. **Принцип работы** — как должно работать пошагово, важные правила и ограничения.
4. **Результат** — что должно произойти/появиться, когда задача решена.

Технические атрибуты (имя, версия, состав скиллов/агентов, модели) скилл решает сам на основе описания.

**Правила активного уточнения:**
- Противоречие в ответах → уточнение с цитатой конфликтующих формулировок.
- Общая формулировка → «приведи конкретный пример».
- Пропущенный граничный случай → «а если [ситуация]?».
- Отсутствие антипаттерна → «а когда плагин НЕ должен срабатывать?».

### `extend-plugin` — скилл (default model)

**Триггер:** свежая сессия, плагин уже существует, пользователь инициирует расширение. Фразы вида «добавь в плагин», «допили скилл», «расширь плагин».

**Флоу:** прочитать целевой плагин → короткое интервью об изменении (какой компонент, что добавить/переработать) → русский конспект → подтверждение → применение (новые файлы или минимальные правки существующих) → версия: MINOR для новых компонентов, PATCH для переработки → git.

### `improve-plugin` — агент (model: opus)

**Триггер:** в текущей сессии только что был инцидент со скиллом/агентом любого плагина. Фразы вида «исправь», «улучши плагин», «запомни это», «так не должно быть», «обнови скилл» и все триггеры из двух существующих `improve-plugin` SKILL-ов.

**Флоу:**

1. Прочитать историю сессии, сформулировать гипотезу о целевом плагине.
2. Первое сообщение с гипотезой-подтверждением: «Вижу проблему в скилле X из плагина Y, правлю его. Если нет — скажи, какой плагин». Пользователь молчит = согласие, пользователь поправляет = переключение.
3. Минимальное точечное исправление в нужном файле. Без рефакторинга смежного кода, без «улучшений» сверх запрошенного.
4. Инкремент PATCH-версии в целевом `plugin.json`.
5. Pre-flight валидация → git → саммари с hash'ем.

Явное ограничение в теле агента: **не вести диалог сверх одного подтверждения плагина**.

### Правило разрешения конфликтов триггеров

Прописано в `description` всех трёх компонентов:

- Если в текущей сессии был инцидент с плагином → `improve-plugin`.
- Если плагин существует, но инцидента не было → `extend-plugin`.
- Если плагина ещё нет → `create-plugin`.

Это устраняет перекрытие триггеров и даёт модели однозначный дискриминатор.

### Shared reference: `references/plugin-authoring.md`

Читается всеми тремя компонентами перед работой. Содержит:

- Структуру `plugin.json`.
- Правила для YAML-frontmatter SKILL.md и agent.md.
- Языковое правило (инструкции English / диалог Russian / коммиты Russian).
- Примеры хороших и плохих триггер-фраз.
- Правила версионирования (PATCH/MINOR/MAJOR с критериями).
- Чек-лист pre-flight валидации.

Вынесено отдельно, чтобы при изменении стандартов генерации правка была в одном месте.

## Валидация и git-флоу

### Pre-flight валидация

Проверки, выполняющиеся перед каждым коммитом. Любой фейл → коммит не делается, русское сообщение об ошибке, файлы остаются на диске для повторного прохода.

| Проверка | Что ловит |
|---|---|
| `plugin.json` парсится как JSON | Синтаксические ошибки генерации |
| `marketplace.json` парсится и не содержит дубль имени | Коллизия имени плагина |
| Все SKILL.md / agent.md непустые | Пропущенная генерация |
| Минимум 3 триггер-фразы у каждого скилла | Размытые триггеры, которые не сработают |
| `version` увеличена (для extend/improve) | Забытый инкремент версии |

### Git-операции

Общая последовательность для всех трёх компонентов:

```
git pull origin main
→ (правка/генерация)
→ (валидация)
git add <файлы по именам, не -A и не .>
git commit -m "<русское сообщение>"
git push
```

Русское саммари после push с hash'ем коммита.

### Rollback

Команды «отмени», «откати», «верни как было» → `git revert <hash>` + push → русское саммари.

## Миграция существующих `improve-plugin`

Разовая операция в рамках реализации `plugin-builder` (делается вручную, а не самим плагином — на момент миграции его ещё нет):

1. Логика из `plugins/anton-toolkit/skills/improve-plugin/SKILL.md` и `plugins/personal-strategist/skills/improve-plugin/SKILL.md` переносится в `plugins/plugin-builder/agents/improve-plugin.md` с параметризацией по плагину (детекция с подтверждением вместо хардкода пути).
2. Два старых SKILL.md удаляются вместе с их директориями.
3. `plugins/anton-toolkit/.claude-plugin/plugin.json` — MINOR-инкремент (удалили компонент).
4. `plugins/personal-strategist/.claude-plugin/plugin.json` — MINOR-инкремент.
5. `.claude-plugin/marketplace.json` — добавлена запись `plugin-builder`.
6. Один коммит на всё с русским сообщением о вынесении `improve-plugin` в отдельный плагин.

Все триггер-фразы из обоих старых SKILL-ов переезжают в `description` агента — ни одна не теряется.

## Структура файлов

```
plugins/plugin-builder/
├── .claude-plugin/
│   └── plugin.json              # version 0.1.0
├── skills/
│   ├── create-plugin/
│   │   └── SKILL.md             # English body, Russian user-facing prompts as templates
│   └── extend-plugin/
│       └── SKILL.md             # same
├── agents/
│   └── improve-plugin.md        # frontmatter: model: opus
└── references/
    └── plugin-authoring.md      # shared generation conventions
```

Корень репозитория: единственное изменение — запись в `.claude-plugin/marketplace.json`.

## Что `plugin-builder` НЕ делает (осознанные ограничения)

- Не переписывает существующие плагины на английский автоматически (миграция языка — точечно, только по запросу через `extend-plugin`).
- Не удаляет плагины целиком (разрушительная операция, вне scope).
- Не трогает плагины вне маркетплейса `anton-toolkit-marketplace`.
- Не генерирует spec-файлы для создаваемых плагинов (дублирует русский конспект из фазы 4 интервью).
- Не пишет тесты (в экосистеме Claude Code плагинов нет формального тестирования).
