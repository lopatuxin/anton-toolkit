## 🚀 Как этим пользоваться

Есть **три способа** запустить инструмент:

### 1. Триггер-фраза (обычный диалог)

Просто пишешь по-русски, Claude сам подберёт скилл:

```
> закоммить изменения                    → запустится commit
> почему API тормозит?                   → запустится debug
> разверни сервис в докере               → запустится devops
> проанализируй мой канал                → запустится yt-my-channel
> спроектируй новый микросервис          → запустится system-designer
```

### 2. Slash-команда (явный вызов)

```
> /commit
> /improve-plugin
> /feature-planner
```

### 3. Делегирование агенту

Агенты вызываются Claude'ом автоматически (если задача под их профиль), либо явно:

```
> вызови code-reviewer на этот PR
> протестируй фичу через qa-engineer
```

> 💡 **Триггеры — не источник истины.** Полный список фраз и нюансов — в YAML-frontmatter каждого `SKILL.md` / `<agent>.md`. Этот README — навигация.

## 📁 Структура маркетплейса

```
anton-toolkit-marketplace/
├── .claude-plugin/
│   └── marketplace.json         # манифест маркетплейса со списком плагинов
├── plugins/                     # все плагины
│   ├── anton-toolkit/
│   ├── personal-strategist/
│   ├── plugin-builder/
│   ├── system-designer/
│   └── youtube-toolkit/
└── docs/
    └── superpowers/             # внутренние материалы
```

### Анатомия одного плагина

```
plugins/<plugin-name>/
├── .claude-plugin/
│   └── plugin.json              # name, version, description
├── skills/                      # скиллы — запускаются в основном диалоге
│   └── <skill-name>/
│       ├── SKILL.md             # инструкции для модели + YAML-frontmatter с триггерами
│       └── references/          # шаблоны, чеклисты, детальные правила
├── agents/                      # subagent'ы — автономные исполнители
│   ├── <agent-name>.md          # системный промпт агента + frontmatter
│   └── references/              # общие референсы
└── commands/                    # slash-команды (если есть)
```

| | **Skills** | **Agents** |
|---|------------|------------|
| **Где работают** | В основном диалоге | В изолированном sub-контексте |
| **Что делают** | Ведут интервью, генерируют документы, оркестрируют | Выполняют одну задачу под ключ |
| **Когда вызывать** | По триггер-фразе или `/<имя>` | Когда нужно делегировать «закрытую» задачу |
| **Пример** | `feature-planner` (интервью + план) | `java-dev` (пишет код по плану) |

## 🛠 anton-toolkit  (v0.18.2)

> **Инженерный швейцарский нож.** Коммиты, ревью, дебаг, devops, dev-агенты под Java/Kotlin/Python/Frontend, генерация тестов и end-to-end QA.

### Skills

<details open>
<summary><b>📝 commit</b> — git-коммиты на русском</summary>

Анализирует staged/unstaged изменения, формирует осмысленное сообщение на русском и делает `git commit`.

**Триггеры:** `"commit"`, `"коммит"`, `"закоммить"`, `"сохрани изменения"`, `/commit`

**Пример:**
```
> закоммить
→ Claude читает diff, генерирует сообщение вида "feat: добавлена валидация email..." и коммитит.
```
</details>

<details>
<summary><b>📄 claude-md</b> — создание/правка CLAUDE.md</summary>

Создаёт, правит, чистит и синхронизирует `CLAUDE.md` под проект с соблюдением лимитов и формата (правила в `references/`).

**Триггеры:** `"claude.md"`, `"создай claude.md"`, `"обнови claude.md"`, `"аудит claude.md"`, `"почисти claude.md"`, `/claude-md`

**Когда использовать:** при первом подключении Claude Code к проекту, после крупных изменений архитектуры, при разрастании файла.
</details>

<details>
<summary><b>🐞 debug</b> — систематический дебаг</summary>

Не угадывает причину — формирует гипотезы, локализует проблему, делает root-cause analysis, чинит.

**Триггеры:** `"баг"`, `"ошибка"`, `"не работает"`, `"падает"`, `"500"`, `"NPE"`, `"тормозит"`, `"зависает"`, `"странное поведение"`, `"почему не работает"`, `/debug`

**Пример:**
```
> у меня в проде 500 на POST /orders, в логах NPE
→ Скилл формирует гипотезы, просит нужные артефакты (stacktrace, код метода), локализует.
```
</details>

<details>
<summary><b>🚢 devops</b> — Docker / деплой / инфра</summary>

Контейнеризация, деплой, инфраструктура. Содержит шаблоны Dockerfile, docker-compose, nginx, CI/CD в `references/`.

**Триггеры:** `"docker"`, `"dockerfile"`, `"compose"`, `"контейнер"`, `"разверни"`, `"задеплой"`, `"nginx"`, `"ci/cd"`, `"пайплайн"`, `"подними в докере"`, `/devops`

**Когда использовать:** упаковка проекта в Docker, написание compose-файла, настройка health-чеков, CI/CD.
</details>

<details>
<summary><b>📋 feature-planner</b> — планирование фичи через интервью</summary>

Проводит интервью по требованиям **ДО** реализации. На выходе — детальный план, по которому потом работают dev-агенты без уточнений.

**Триггеры:** `"спланируй фичу"`, `"распланируй фичу"`, `"продумай фичу"`, `"спроектируй фичу"`, `"уточни нюансы"`, `"вот тз"`, `/feature-planner`

> ⚠️ Не путать с `system-designer`: тот проектирует **систему целиком** (документация), а `feature-planner` готовит **одну фичу к имплементации**.

**Когда использовать:** перед стартом любой нетривиальной фичи — чтобы dev-агенты потом не задавали лишних вопросов.
</details>

<details>
<summary><b>🔧 improve-plugin</b> — точечная правка скилла/агента в этом маркетплейсе</summary>

Реактивная правка после инцидента: пользователь говорит «вот эта штука глючит, исправь» — скилл правит исходник в `plugins/`, бампит PATCH-версию, делает git push.

**Триггеры (только явные):** `"улучши плагин"`, `"обнови скилл"`, `"исправь агента"`, `"запомни это в плагине"`, `/improve-plugin`

> ⚠️ Не вызывается проактивно. Только когда пользователь явно попросил.

**Альтернативы:** для нового плагина — `create-plugin`, для расширения существующего — `extend-plugin` (оба в `plugin-builder`).
</details>

### Agents

| Агент | Когда сработает | Что делает |
|-------|-----------------|------------|
| 🟧 **java-dev** | Файл внутри Java-модуля (`pom.xml` в дереве) | Пишет/правит `.java`, SQL, Spring XML, Velocity, HTML/CSS/JS в `src/main/`. **Триггер — модуль, не расширение.** |
| 🟪 **kotlin-dev** | Файл внутри Kotlin-модуля (`build.gradle.kts`) | Пишет/правит `.kt`, `.kts`, SQL, Liquibase XML, YAML-конфиги. |
| 🐍 **python-dev** | Файл внутри Python-модуля (`pyproject.toml`/`setup.py`/`requirements.txt`) | Реализует по step-by-step плану: `.py`, Alembic, Jinja, Dockerfile, конфиги. |
| 🌐 **frontend-dev** | `*.tsx`, `*.ts`, `*.jsx`, `*.js`, `*.css`, `*.scss`, `*.html`, ассеты в `public/` | Любая фронтовая правка — даже одна строчка идёт через агента. |
| 👀 **code-reviewer** | `"проверь код"`, `"сделай ревью"`. Автоматически после dev-агентов (по правилу в `CLAUDE.md`) | Структурированный отчёт: баги, безопасность, нарушения паттернов. Java, Kotlin, TS, React, CSS, SQL, Dockerfile. |
| 🧪 **test-writer** | Проактивно после dev-агентов; явно — `"напиши тесты для X"` | Читает класс, находит существующие паттерны тестов, пишет в том же стиле. |
| 🕵 **qa-engineer** | `"протестируй фичу"`, `"smoke-тест"`, `"протестируй перед мержем"` | End-to-end тест: API через `curl`, UI через Chrome DevTools, выдаёт bug-report с маршрутизацией (backend bug → backend dev). |

## 🎯 personal-strategist  (v0.3.0)

> Личный стратегический советник. Собирает информацию о тебе, систематизирует, помогает строить стратегии достижения целей. Данные хранятся в отдельном репозитории-профиле.

### Skills

<details open>
<summary><b>👋 onboarding</b> — первичное знакомство</summary>

Первый раз запускаешь личного стратега? Этот скилл проведёт интервью: цели, контекст, ограничения, ценности. Создаёт `personal-profile` репозиторий.

**Триггеры:** `"познакомимся"`, `"расскажу о себе"`, `"онбординг"`, `"создай профиль"`, `"начнём знакомство"`

**Особенность:** интервью идёт прямо в диалоге, без делегирования агентам (агент терял бы контекст между turns).
</details>

<details>
<summary><b>🔍 profile-review</b> — ревью и обновление профиля</summary>

Полный аудит существующего профиля: пробелы, устаревшие данные, уточнения.

**Триггеры:** `"ревью профиля"`, `"обнови профиль"`, `"проверь данные обо мне"`, `"что ты знаешь обо мне"`, `"актуализируй профиль"`

**Когда использовать:** раз в квартал или после крупных изменений в жизни/работе.
</details>

<details>
<summary><b>♟ strategist</b> — стратегические рекомендации</summary>

На основе профиля выдаёт стратегию, план действий, рекомендации.

**Триггеры:** `"стратегия"`, `"как мне достичь"`, `"план действий"`, `"что делать с"`, `"посоветуй"`, `"как поступить"`, `"стратег"`

**Требование:** профиль должен существовать (если нет — сначала `onboarding`).
</details>

### Agents

| Агент | Роль |
|-------|------|
| 🎤 **interviewer** | Ведёт интервью, аккуратно вытягивает информацию. |
| 📊 **analyst** | Анализирует собранные данные, выявляет паттерны. |
| 👁 **observer** | Фиксирует наблюдения о поведении и предпочтениях. |
| ♟ **strategist** | Формирует стратегические рекомендации. |

## 🏗 plugin-builder  (v0.2.0)

> Конструктор плагинов для этого маркетплейса. Делает то же, что `improve-plugin`, только **проактивно** — когда хочешь добавить новое, а не починить старое.

### Skills

<details open>
<summary><b>✨ create-plugin</b> — создание плагина с нуля</summary>

Интервью → структура → `plugin.json` → первые скиллы/агенты → регистрация в `marketplace.json`.

**Триггеры:** `"создай плагин"`, `"нужен новый плагин"`, `"хочу плагин для"`, `"сделай плагин, который"`, `"создай инструмент"`

**Когда использовать:** когда нужного плагина ещё нет в `plugins/`.
</details>

<details>
<summary><b>➕ extend-plugin</b> — расширение существующего плагина</summary>

Добавляет новый скилл/агент/референс в уже существующий плагин **в свежей сессии** (когда не было инцидента — иначе используется `improve-plugin`).

**Триггеры:** `"добавь в плагин"`, `"допили скилл"`, `"расширь плагин"`, `"хочу в <name> добавить"`, `"нужно дополнить плагин"`

**Дискриминация:**
- Плагина ещё нет → `create-plugin`
- Плагин есть, инцидент в текущей сессии → `improve-plugin` (из `anton-toolkit`)
- Плагин есть, свежая сессия, хочу больше функционала → `extend-plugin`
</details>

## 📐 system-designer  (v0.3.4)

> Итеративное проектирование софтверных систем. Производит документацию в `docs/` репозитория (concept → architecture → modules → roadmap → phase details). **Код не пишет.**

### Skill

<details open>
<summary><b>📐 system-designer</b> — оркестратор всех 6 фаз</summary>

Один скилл управляет всем процессом, на каждой фазе делегирует соответствующему агенту.

**Триггеры:** `"спроектируй систему"`, `"задизайним модуль"`, `"опиши модуль X"`, `"разбей на фазы"`, `"сделай roadmap"`, `"построй план фаз"`, `"детализируй фазу"`, `"продолжим проектировать"`
</details>

### Agents (по фазам)

```
Фаза 1 (интервью)    →  идёт в диалоге, без агента
       │
Фаза 2 (architecture) →  📐 architect       → docs/architecture.md
       │
Фаза 3 (modules)      →  🧩 module-designer  → docs/modules/<name>.md (один на модуль)
       │
Фаза 4 (изменения)    →  ✏️ docs-updater    → пропагирует правку по всем затронутым докам
       │
Фаза 5 (roadmap)      →  🗺 roadmap-planner  → docs/roadmap.md
       │
Фаза 6 (детализация)  →  🔍 phase-detailer   → docs/phases/phase-NN-<slug>.md
```

| Агент | Что пишет |
|-------|-----------|
| 📐 **architect** | `docs/architecture.md` — компоненты, интерфейсы, data flow, стек с обоснованием. |
| 🧩 **module-designer** | `docs/modules/<name>.md` — ответственности, интерфейсы, data model, потоки, ошибки. |
| ✏️ **docs-updater** | Когда дизайн меняется — пропагирует изменение по `concept.md`, `architecture.md`, `modules/*.md`. |
| 🗺 **roadmap-planner** | `docs/roadmap.md` — фазы реализации, end-to-end слайсы, зависимости. |
| 🔍 **phase-detailer** | `docs/phases/phase-NN-<slug>.md` — детально на фазу, чтобы `python-dev` мог реализовать без вопросов. |
| 🧐 **doc-reviewer** | Не пишет — **ревьюит** документы, созданные другими агентами. Вызывается оркестратором после каждого write-шага во всех фазах: проверка на шаблон, согласованность с `docs/`, нарушения YAGNI/KISS/LLM-friendliness. |

## 🎬 youtube-toolkit  (v0.3.3)

> Полный набор для ведения русскоязычного IT/vibe-coding YouTube-канала.
> **Cadence:** 1 видео/неделя, ≤ 20 минут, **без Shorts**.

> 🆕 **v0.3.x:** появилась персистентная **база знаний канала** (`yt-knowledge-base`) — Obsidian-vault со всеми видео, транскриптами, метриками и классификацией top/fail. Остальные скиллы (`yt-ideas`, `yt-script`, `yt-seo`, `yt-thumbnail`, `yt-promo`, `yt-content-plan`) подтягивают её как контекст автоматически. Поверх базы работает агент-стратег `yt-strategist`, выдающий топ-5 идей на неделю.

### Skills

<details open>
<summary><b>🗄 yt-knowledge-base</b> — персистентная база знаний канала</summary>

Собирает и хранит на диске **полный датасет** по каналу: карточка каждого видео (metadata + транскрипт + классификация top/fail), который автоматически подгружается остальными yt-скиллами как контекст.

**Триггеры:** `"собери базу по моему каналу"`, `"собери базу знаний канала"`, `"обнови базу видео"`, `"обнови БД канала"`, `"база знаний канала"`, `"построй базу канала"`, `"пересобери базу видео"`

**Два режима:**
- **INIT** — полная сборка с нуля (все видео канала)
- **REFRESH** — инкрементально: новые видео + обновлённые метрики по существующим

**Когда использовать:** один раз для INIT, потом периодически REFRESH (например, раз в неделю перед планированием).
</details>

<details>
<summary><b>📊 yt-my-channel</b> — анализ собственного канала</summary>

Что работает на канале, что не работает, топ-видео, метрики, текущая динамика. Умеет парсить вставленные данные из YouTube Studio.

**Триггеры:** `"проанализируй мой канал"`, `"топ видео на моём канале"`, `"статистика канала"`, `"метрики моего канала"`, `"оцени мой канал"`

**Не путать с** `yt-competitors` — это про себя.
</details>

<details>
<summary><b>🔭 yt-competitors</b> — конкуренты и ниша</summary>

Два режима:
- **SINGLE** — разбор одного канала глубоко
- **BATCH** — скан ниши по каналам выше порога подписчиков

**Триггеры:** `"анализ конкурентов"`, `"анализ ниши"`, `"разбери канал X"`, `"что заходит в нише"`, `"сравни с конкурентами"`, `"тренды в нише"`
</details>

<details>
<summary><b>💡 yt-ideas</b> — генерация тем для видео</summary>

**⚠️ Hard requirement:** скилл откажется работать без отчёта от `yt-my-channel` или `yt-competitors` в контексте. Сначала собираем данные, потом идеи.

**Триггеры:** `"идеи для роликов"`, `"о чём снять"`, `"темы для видео"`, `"что снять следующим"`
</details>

<details>
<summary><b>📅 yt-content-plan</b> — контент-план / календарь</summary>

Календарь публикаций с учётом cadence (1 видео/неделю).

**Триггеры:** `"контент-план"`, `"расписание видео"`, `"календарь канала"`, `"контент на месяц"`, `"график выхода видео"`
</details>

<details>
<summary><b>🎬 yt-script</b> — сценарий ролика</summary>

Hook, акты, CTA, темп. Максимум 20 минут.

**Триггеры:** `"сценарий для видео"`, `"напиши сценарий"`, `"hook для видео"`, `"структура ролика"`, `"скрипт ролика"`
</details>

<details>
<summary><b>🔎 yt-seo</b> — название, описание, теги</summary>

Заголовок (A/B варианты), описание с таймкодами, теги.

**Триггеры:** `"название и описание"`, `"тайтл и теги"`, `"подбери название"`, `"опиши видео"`, `"теги для видео"`

**Что НЕ делает:** превью — это `yt-thumbnail`.
</details>

<details>
<summary><b>🖼 yt-thumbnail</b> — концепт превью</summary>

Идея превью: текст, композиция, цвет, эмоция. **Картинку не рисует** — это концепт, который ты потом нарисуешь сам или в инструменте.

**Триггеры:** `"превью для видео"`, `"обложка для видео"`, `"thumbnail концепт"`, `"что нарисовать на превью"`
</details>

<details>
<summary><b>🎯 yt-promo</b> — продвижение и лидген на канале</summary>

**On-channel only:** pinned-комменты, end-screens, CTA в видео и описании, воронка на сайт-визитку.
**Out of scope:** кросс-промо на других платформах (Telegram, Habr и т.п.).

**Триггеры:** `"продвижение канала"`, `"как раскрутить канал"`, `"лидген на канале"`, `"привлечь заказы"`, `"закреп под видео"`, `"end screen для видео"`
</details>

## 🛠 Поддержка и развитие

### Где править

| Что | Чем |
|-----|-----|
| Новый плагин | `/create-plugin` (из `plugin-builder`) |
| Новый скилл/агент в существующем плагине | `/extend-plugin` (из `plugin-builder`) |
| Точечная правка скилла/агента после инцидента | `/improve-plugin` (из `anton-toolkit`) |

> ⚠️ **Не редактируй** руками файлы в `~/.claude/plugins/cache/` — это read-only кэш, изменения затрутся. Реальный источник — этот репозиторий, путь зависит от машины. На текущей машине: `~/.claude/plugins/marketplaces/anton-toolkit-marketplace/` (он же — git working copy).

### Версионирование

Каждая правка плагина → **bump PATCH** в `plugins/<name>/.claude-plugin/plugin.json`. Без bump'а маркетплейс не подтянет изменение на других машинах.
