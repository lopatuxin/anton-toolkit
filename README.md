## 🚀 Как этим пользоваться

Есть **три способа** запустить инструмент:

### 1. Триггер-фраза (обычный диалог)

Просто пишешь по-русски, Claude сам подберёт скилл:

```
> закоммить изменения                    → запустится commit
> почему API тормозит?                   → запустится debug
> разверни сервис в докере               → запустится devops
> проанализируй мой канал                → запустится yt-my-channel
> спроектируй новый микросервис          → запустится system-designer
```

> ⚠️ Часть скиллов (личные и курсовые — `dnevnik`, `film`, `idea`, `itogi`, `profil`, `lesson`, `roadmap`) **запускаются только по slash-команде**, без авто-триггера.

### 2. Slash-команда (явный вызов)

```
> /commit
> /improve-plugin
> /feature-planner
> /dnevnik
> /logos-design
```

### 3. Делегирование агенту

Агенты вызываются Claude'ом автоматически (если задача под их профиль), либо явно:

```
> вызови code-reviewer на этот PR
> протестируй фичу через qa-engineer
```

> 💡 **Триггеры — не источник истины.** Полный список фраз и нюансов — в YAML-frontmatter каждого `SKILL.md` / `<agent>.md`. Этот README — навигация.

```
│   ├── plugin-builder/

└── docs/
    └── superpowers/             # внутренние материалы
```

### Анатомия одного плагина

```
plugins/<plugin-name>/
├── .claude-plugin/
│   └── plugin.json              # name, version, description
├── skills/                      # скиллы — запускаются в основном диалоге
│   └── <skill-name>/
│       ├── SKILL.md             # инструкции для модели + YAML-frontmatter с триггерами
│       └── references/          # шаблоны, чеклисты, детальные правила
├── agents/                      # subagent'ы — автономные исполнители
│   ├── <agent-name>.md          # системный промпт агента + frontmatter
│   └── references/              # общие референсы
└── commands/                    # slash-команды (если есть)
```

| | **Skills** | **Agents** |
|---|------------|------------|
| **Где работают** | В основном диалоге | В изолированном sub-контексте |
| **Что делают** | Ведут интервью, генерируют документы, оркестрируют | Выполняют одну задачу под ключ |
| **Когда вызывать** | По триггер-фразе или `/<имя>` | Когда нужно делегировать «закрытую» задачу |
| **Пример** | `feature-planner` (интервью + план) | `java-dev` (пишет код по плану) |

## 🛠 anton-toolkit

> **Инженерный швейцарский нож.** Коммиты, ревью, дебаг, devops, dev-агенты под Java/Kotlin/Python/Frontend, генерация тестов и end-to-end QA.

### Skills

<details open>
<summary><b>📝 commit</b> — git-коммиты на русском</summary>

Анализирует staged/unstaged изменения, формирует осмысленное сообщение на русском и делает `git commit`.

**Триггеры:** `"commit"`, `"коммит"`, `"закоммить"`, `"сохрани изменения"`, `/commit`

**Пример:**
```
> закоммить
→ Claude читает diff, генерирует сообщение вида "feat: добавлена валидация email..." и коммитит.
```
</details>

<details>
<summary><b>📄 claude-md</b> — создание/правка CLAUDE.md</summary>

Создаёт, правит, чистит и синхронизирует `CLAUDE.md` под проект с соблюдением лимитов и формата (правила в `references/`).

**Триггеры:** `"claude.md"`, `"создай claude.md"`, `"обнови claude.md"`, `"аудит claude.md"`, `"почисти claude.md"`, `/claude-md`

**Когда использовать:** при первом подключении Claude Code к проекту, после крупных изменений архитектуры, при разрастании файла.
</details>

<details>
<summary><b>🐞 debug</b> — систематический дебаг</summary>

Не угадывает причину — формирует гипотезы, локализует проблему, делает root-cause analysis, чинит.

**Триггеры:** `"баг"`, `"ошибка"`, `"не работает"`, `"падает"`, `"500"`, `"NPE"`, `"тормозит"`, `"зависает"`, `"странное поведение"`, `"почему не работает"`, `/debug`

**Пример:**
```
> у меня в проде 500 на POST /orders, в логах NPE
→ Скилл формирует гипотезы, просит нужные артефакты (stacktrace, код метода), локализует.
```
</details>

<details>
<summary><b>🚢 devops</b> — Docker / деплой / инфра</summary>

Контейнеризация, деплой, инфраструктура. Содержит шаблоны Dockerfile, docker-compose, nginx, CI/CD в `references/`.

**Триггеры:** `"docker"`, `"dockerfile"`, `"compose"`, `"контейнер"`, `"разверни"`, `"задеплой"`, `"nginx"`, `"ci/cd"`, `"пайплайн"`, `"подними в докере"`, `/devops`

**Когда использовать:** упаковка проекта в Docker, написание compose-файла, настройка health-чеков, CI/CD.
</details>

<details>
<summary><b>📋 feature-planner</b> — планирование фичи через интервью</summary>

Проводит интервью по требованиям **ДО** реализации. На выходе — детальный план, по которому потом работают dev-агенты без уточнений.

**Триггеры:** `"спланируй фичу"`, `"распланируй фичу"`, `"продумай фичу"`, `"спроектируй фичу"`, `"уточни нюансы"`, `"вот тз"`, `/feature-planner`

> ⚠️ Не путать с `system-designer`: тот проектирует **систему целиком** (документация), а `feature-planner` готовит **одну фичу к имплементации**.

**Когда использовать:** перед стартом любой нетривиальной фичи — чтобы dev-агенты потом не задавали лишних вопросов.
</details>

### Agents

| Агент | Когда сработает | Что делает |
|-------|-----------------|------------|
| 🟧 **java-dev** | Файл внутри Java-модуля (`pom.xml` в дереве) | Пишет/правит `.java`, SQL, Spring XML, Velocity, HTML/CSS/JS в `src/main/`. **Триггер — модуль, не расширение.** |
| 🟪 **kotlin-dev** | Файл внутри Kotlin-модуля (`build.gradle.kts`) | Пишет/правит `.kt`, `.kts`, SQL, Liquibase XML, YAML-конфиги. |
| 🐍 **python-dev** | Файл внутри Python-модуля (`pyproject.toml`/`setup.py`/`requirements.txt`) | Реализует по step-by-step плану: `.py`, Alembic, Jinja, Dockerfile, конфиги. |
| 🌐 **frontend-dev** | `*.tsx`, `*.ts`, `*.jsx`, `*.js`, `*.css`, `*.scss`, `*.html`, ассеты в `public/` | Любая фронтовая правка — даже одна строчка идёт через агента. |
| 👀 **code-reviewer** | `"проверь код"`, `"сделай ревью"`. Автоматически после dev-агентов (по правилу в `CLAUDE.md`) | Структурированный отчёт: баги, безопасность, нарушения паттернов. Java, Kotlin, TS, React, CSS, SQL, Dockerfile. |
| 📝 **mr-spec-reviewer** | Только командой `/mr-review` — без триггер-фраз | Сверяет всю ветку (MR) с твоей документацией из `project/documentation/` и выдаёт готовые тексты замечаний для MR: `файл:строка` + короткое сообщение без жаргона. Ищет и баги/безопасность, и нарушения контрактов из доки; дока же отсеивает ложные срабатывания. |
| 🧪 **test-writer** | Проактивно после dev-агентов; явно — `"напиши тесты для X"` | Читает класс, находит существующие паттерны тестов, пишет в том же стиле. |
| 🕵 **qa-engineer** | `"протестируй фичу"`, `"smoke-тест"`, `"протестируй перед мержем"` | End-to-end тест: API через `curl`, UI через Chrome DevTools, выдаёт bug-report с маршрутизацией (backend bug → backend dev). |

## 📓 lichnost

> Личное логирование жизни в Obsidian-vault: дневник, идеи, фильмы, аналитические итоги и эволюционирующий психологический самопрофиль. Все скиллы вызываются **только по slash-команде** — без авто-триггера. Анализ и капча, кода не пишет.

### Skills

<details open>
<summary><b>📔 dnevnik</b> — дневник под диктовку</summary>

Записывает дневниковую запись в Obsidian-vault, форматируя сырые мысли в чистый Markdown-нот с именем по дате. Только капча и форматирование — без анализа.

**Вызов:** `/dnevnik`
</details>

<details>
<summary><b>💭 idea</b> — захват и развитие идей</summary>

Каждая идея — отдельный нот под `Личная/Идеи`, написанный как лёгкая документация (переформулировка для ясности, без выдумок). Два потока: создать новую идею или дополнить существующую.

**Вызов:** `/idea`
</details>

<details>
<summary><b>🎬 film</b> — личный кинокуратор</summary>

Ведёт три нота: профиль вкуса, лог просмотренного с рецензиями и короткий шортлист рекомендаций (≤10 фильмов). Три потока: порекомендовать, залогировать просмотренный фильм, мигрировать старую заметку.

**Вызов:** `/film`
</details>

<details>
<summary><b>📈 itogi</b> — аналитические итоги периода</summary>

Каскадный разбор дневника: недельный итог анализирует дневные записи, месячный — недельные итоги, годовой — месячные. Анализ, а не плоская сводка.

**Вызов:** `/itogi`
</details>

<details>
<summary><b>🪞 profil</b> — психологический самопрофиль</summary>

Строит и развивает глубокий профиль личности из данных vault (дневник, итоги, идеи) плюс адаптивные интервью. Два режима: ИНТЕРВЬЮ (живой диалог по одной теме) и ОБНОВИ/АНАЛИЗ (пересборка профиля по всем данным).

**Вызов:** `/профиль` (`/profil`)
</details>

## 🧠 logos

> Инструментарий для проектирования **и разработки** **Logos** — автономного ИИ-ассистента (личный «Джарвис»). Доменно-специфичный **совет архитекторов** проектирует систему на одном общем черновике с журналом обсуждения, плюс поисковый журнал решений с весами важности. Поверх дизайна работает **команда разработки**: оркестратор `logos-build` гонит фазу через свою команду выделенных агентов (coder → reviewer → test → qa → devops → sync), превращая документацию в код. Документация живёт в папке `Logos/` Obsidian-vault (auto-sync), код — в отдельном репозитории `git@github.com:lopatuxin/Logos.git`; они всегда синхронизированы, источник правды — документация. Код пишется расширяемым и понятным для ИИ-агентов, а не для людей.

### Skills

<details open>
<summary><b>🏛 logos-design</b> — совет архитекторов Logos</summary>

Проектирует систему Logos через совет из пяти членов (оркестрация, память, модели, автономия, ресурсный реализм): лид кладёт скелет, каждый углубляет свой домен и спорит с остальными, раунд разрешения закрывает открытые вопросы, синтезатор консолидирует converged-черновик в одну каноническую архитектуру. Требования собираются в диалоге, каждое ключевое решение ревьюится пользователем и пишется в журнал.

**Триггеры:** `"спроектируй logos"`, `"архитектура logos"`, `"созови совет по logos"`, `"продолжим проектировать logos"`, `/logos-design`

> ⚠️ Только для проекта Logos с фиксированным советом. Для проектирования произвольных систем — `system-designer`.
</details>

<details>
<summary><b>🗒 logos-log</b> — журнал решений</summary>

Поисковый журнал, где каждое решение и эксперимент записаны с весом важности и ревью пользователя.

**Вызов:** `/logos-log`
</details>

<details>
<summary><b>🖥 logos-ui</b> — спека веб-интерфейса под Claude Design</summary>

Прорабатывает структуру web-интерфейса Logos (страницы, навигация, блоки, кнопки, элементы, состояния, флоу) как build-ready спеку для Claude Design — **без визуала и цветов** (их выбирает сам Claude Design). Ведёт глубокое интервью (открытые вопросы по одному), пишет спеку в `Logos/Дизайн/Веб-интерфейс.md`, после правок **синхронизирует** её с `Архитектура.md` и пишет значимые изменения в журнал.

**Вызов:** `/logos-ui` (только командой, без авто-триггера)
</details>

<details>
<summary><b>🧩 logos-phases</b> — нарезка архитектуры на фазы</summary>

Режет систему из `Архитектура.md` на инкрементальные **фазы поставки**: каждая — законченный сквозной кусок функционала, который можно пощупать, запустить и протестировать (старт с MVP-ноль, дальше по одной осязаемой возможности). Ведёт глубокое интервью (открытые вопросы по одному), пишет **по одному markdown-файлу на фазу** в `Logos/Дизайн/Фазы/Фаза-NN-<имя>.md` — **без дорожной карты** (обзор ведётся в журнале), каждую фазу пишет в журнал и **сразу синхронизирует** документацию (правит `Архитектура.md`/`Концепт.md` при рассинхроне).

**Вызов:** `/logos-phases` (только командой, без авто-триггера)
</details>

<details>
<summary><b>🏗 logos-build</b> — разработка Logos по фазам (оркестратор команды)</summary>

Единственный скилл Logos, который **пишет реальный код**. Берёт фазу из `Logos/Дизайн/Фазы/` как ТЗ (документация — источник правды), при первом запуске сам клонирует/инициализирует код-репо `git@github.com:lopatuxin/Logos.git` (соседняя с vault папка `Logos/`), затем гонит фазу через команду выделенных Logos-агентов: **coder → reviewer → test → qa → devops → sync**, обновляет статус фазы и пишет запись в журнал. Полиглот: каждый слой — на своём стеке по архитектуре. Код пишется **расширяемым и понятным для ИИ-агентов, а не для людей** (хозяин в код не лезет). Использует только `logos-*` агентов, не anton-toolkit (те заточены под рабочие проекты).

**Триггеры:** `"собери logos"`, `"реализуй фазу logos"`, `"запили фазу logos"`, `"разработай logos"`, `"продолжи разработку logos"`, `/logos-build`

> ⚠️ Дизайн (без кода) — `logos-design`/`logos-phases`/`logos-ui`. Только этот скилл порождает рабочий код.
</details>

| 🧭 **logos-orchestration-architect** | **Лид.** Оркестрация: иерархия центрального «мозга» → блок-оркестраторы → рои агентов. Кладёт базовый скелет. |
| 🧠 **logos-memory-engineer** | Память с весами важности/силы, ночная консолидация, сила растёт на успехе и падает на неудаче. |
| 🤖 **logos-model-engineer** | Рой маленьких специализированных моделей, путь от готовых (китайские через OpenRouter) к локальным, роутинг задача→модель, fine-tuning. |
| 🛠 **logos-autonomy-architect** | Автономия: система создаёт свои инструменты и скиллы, саморегистрация, самомодификация и её границы безопасности. |
| 💰 **logos-resource-realist** | Ресурсный реализм: всё работает на скромном железе (одна рабочая станция, ~72 GB VRAM), без датацентров. Идёт последним — реагирует на всех. |
| ✍️ **logos-synthesizer** | Закрывает совет: читает converged-черновик и журнал обсуждения, полирует в финальный документ архитектуры. |

### Agents (команда разработки)

Дирижируются скиллом `logos-build`, не вызываются напрямую. Все несут контекст проекта и доктрину «код для ИИ, не для людей» (`references/logos-project.md`).

| Агент | Что делает |
|-------|-----------|
| ⌨️ **logos-coder** | Пишет код Logos по фазе. Полиглот (стек под слой по архитектуре), код явный, с машинно-читаемыми манифестами, расширяемый регистрацией. |
| 👁 **logos-reviewer** | Ревью против **двух** эталонов: архитектурных доков и доктрины. Отклоняет «человеческие» упрощения, снижающие явность/машинную читаемость. |
| 🧪 **logos-test-writer** | Тесты как машинно-проверяемая спека: «Критерии готовности» фазы → ассерты, фреймворк под стек каждого слоя. |
| 🕵 **logos-qa** | Сквозная проверка фазы по «Критериям готовности» (API + веб-UI через браузер + панель логов + пути ошибок), bug-report с маршрутизацией. |
| 🚢 **logos-devops** | Делает фазу запускаемой в рамках ресурсного бюджета: контейнеры/скрипты/конфиг, секреты вне репо, health-чек. |
| 🔄 **logos-sync** | Сверяет код с документацией в обе стороны, докладывает каждый drift (`file:line` ↔ раздел доки). Сам ничего не меняет. |

## 🏗 plugin-builder

> Конструктор плагинов для этого маркетплейса. `create-plugin` и `extend-plugin` работают **проактивно** (создать новое / расширить), `improve-plugin` — **реактивно** (точечная правка после инцидента).

### Skills

<details open>
<summary><b>✨ create-plugin</b> — создание плагина с нуля</summary>

Интервью → структура → `plugin.json` → первые скиллы/агенты → регистрация в `marketplace.json`.

**Триггеры:** `"создай плагин"`, `"нужен новый плагин"`, `"хочу плагин для"`, `"сделай плагин, который"`, `"создай инструмент"`

**Когда использовать:** когда нужного плагина ещё нет в `plugins/`.
</details>

<details>
<summary><b>➕ extend-plugin</b> — расширение существующего плагина</summary>

Добавляет новый скилл/агент/референс в уже существующий плагин **в свежей сессии** (когда не было инцидента — иначе используется `improve-plugin`).

**Триггеры:** `"добавь в плагин"`, `"допили скилл"`, `"расширь плагин"`, `"хочу в <name> добавить"`, `"нужно дополнить плагин"`

**Дискриминация:**
- Плагина ещё нет → `create-plugin`
- Плагин есть, инцидент в текущей сессии → `improve-plugin`
- Плагин есть, свежая сессия, хочу больше функционала → `extend-plugin`
</details>

<details>
<summary><b>🔧 improve-plugin</b> — точечная правка скилла/агента в этом маркетплейсе</summary>

Реактивная правка после инцидента: пользователь говорит «вот эта штука глючит, исправь» — скилл правит исходник в `plugins/`, бампит PATCH-версию, делает git push.

**Триггеры (только явные):** `"улучши плагин"`, `"обнови скилл"`, `"исправь агента"`, `"запомни это в плагине"`, `/improve-plugin`

> ⚠️ Не вызывается проактивно. Только когда пользователь явно попросил.

**Альтернативы:** для нового плагина — `create-plugin`, для расширения существующего — `extend-plugin`.
</details>

## 📐 system-designer

> Итеративное проектирование софтверных систем. Производит документацию в `Документация/` или `docs/` репозитория (концепт → архитектура → модули → управление изменениями → roadmap → детализация фаз). **Код не пишет.**

### Skill

<details open>
<summary><b>📐 system-designer</b> — оркестратор всех фаз</summary>

Один скилл управляет всем процессом, на каждой фазе делегирует соответствующему агенту. Интервью идёт в диалоге; агенты дозреваются только на автономные write-шаги.

**Триггеры:** `"спроектируй систему"`, `"задизайним модуль"`, `"опиши модуль X"`, `"разбей на фазы"`, `"сделай roadmap"`, `"построй план фаз"`, `"детализируй фазу"`, `"продолжим проектировать"`
</details>

### Agents (по фазам)

```
Фаза 1 (интервью)        →  идёт в диалоге, без агента
       │
Фаза 2 (архитектура)     →  🏛 consortium-specialist (совет на общем черновике)
       │                     ↳ ✍️ architecture-synthesizer → финальная архитектура
       │
Фаза 3 (модули)          →  🧩 module-designer  → Модули/<имя>.md (один на модуль)
       │
Фаза 4 (изменения)       →  ✏️ docs-updater     → пропагирует правку по всем затронутым докам
       │
Фаза 5 (roadmap)         →  🗺 roadmap-planner   → Дорожная карта.md
       │
Фаза 6 (детализация)     →  🔍 phase-detailer    → Фазы/Фаза-NN-<имя>.md
```

| Агент | Что делает |
|-------|-----------|
| 🏛 **consortium-specialist** | Один член совещательного консорциума на **общем** черновике архитектуры: параметризуется ролью (lead/data/security/integrations/infra/perf/domain) и режимом (skeleton/contribute/resolve). Углубляет свой домен, поднимает вопросы к остальным. |
| ✍️ **architecture-synthesizer** | Лид-архитектор, закрывающий совет: полирует converged-черновик в единый каноничный документ архитектуры и выдаёт сводку ключевых решений. |
| 🧩 **module-designer** | `Модули/<имя>.md` — ответственности, интерфейсы, data model, потоки, ошибки, специфика стека. |
| ✏️ **docs-updater** | Когда дизайн меняется — пропагирует изменение по концепту, архитектуре, модулям. |
| 🗺 **roadmap-planner** | `Дорожная карта.md` — фазы реализации, end-to-end слайсы, зависимости. |
| 🔍 **phase-detailer** | `Фазы/Фаза-NN-<имя>.md` — детально на фазу, чтобы dev-агент мог реализовать без вопросов. |
| 🧐 **doc-reviewer** | Не пишет — **ревьюит** документы после каждого write-шага: проверка на шаблон, согласованность, нарушения YAGNI/KISS/LLM-friendliness. |

## 🎬 youtube-toolkit

> Полный набор для ведения русскоязычного IT/vibe-coding YouTube-канала.
> **Cadence:** 1 видео/неделя, ≤ 20 минут, **без Shorts**.

> 🆕 **v0.3.x+:** персистентная **база знаний канала** (`yt-knowledge-base`) — Obsidian-vault со всеми видео, транскриптами, метриками и классификацией top/fail. Остальные скиллы подтягивают её как контекст автоматически. Поверх базы работает агент-стратег `yt-strategist`, выдающий топ-5 идей на неделю. В v0.4.x добавлен `yt-montage` — монтажный лист из готового сценария.

### Skills

<details open>
<summary><b>🗄 yt-knowledge-base</b> — персистентная база знаний канала</summary>

Собирает и хранит на диске **полный датасет** по каналу: карточка каждого видео (metadata + транскрипт + классификация top/fail), который автоматически подгружается остальными yt-скиллами как контекст.

**Триггеры:** `"собери базу по моему каналу"`, `"собери базу знаний канала"`, `"обнови базу видео"`, `"обнови БД канала"`, `"база знаний канала"`, `"построй базу канала"`, `"пересобери базу видео"`

**Два режима:**
- **INIT** — полная сборка с нуля (все видео канала)
- **REFRESH** — инкрементально: новые видео + обновлённые метрики по существующим

**Когда использовать:** один раз для INIT, потом периодически REFRESH (например, раз в неделю перед планированием).
</details>

<details>
<summary><b>📊 yt-my-channel</b> — анализ собственного канала</summary>

Что работает на канале, что не работает, топ-видео, метрики, текущая динамика. Умеет парсить вставленные данные из YouTube Studio.

**Триггеры:** `"проанализируй мой канал"`, `"топ видео на моём канале"`, `"статистика канала"`, `"метрики моего канала"`, `"оцени мой канал"`

**Не путать с** `yt-competitors` — это про себя.
</details>

<details>
<summary><b>🔭 yt-competitors</b> — конкуренты и ниша</summary>

Два режима:
- **SINGLE** — разбор одного канала глубоко
- **BATCH** — скан ниши по каналам выше порога подписчиков

**Триггеры:** `"анализ конкурентов"`, `"анализ ниши"`, `"разбери канал X"`, `"что заходит в нише"`, `"сравни с конкурентами"`, `"тренды в нише"`
</details>

<details>
<summary><b>💡 yt-ideas</b> — генерация тем для видео</summary>

**⚠️ Hard requirement:** скилл откажется работать без отчёта от `yt-my-channel` или `yt-competitors` в контексте. Сначала собираем данные, потом идеи.

**Триггеры:** `"идеи для роликов"`, `"о чём снять"`, `"темы для видео"`, `"что снять следующим"`
</details>

<details>
<summary><b>📅 yt-content-plan</b> — контент-план / календарь</summary>

Календарь публикаций с учётом cadence (1 видео/неделю).

**Триггеры:** `"контент-план"`, `"расписание видео"`, `"календарь канала"`, `"контент на месяц"`, `"график выхода видео"`
</details>

<details>
<summary><b>🎬 yt-script</b> — сценарий ролика</summary>

Hook, акты, CTA, темп. Максимум 20 минут.

**Триггеры:** `"сценарий для видео"`, `"напиши сценарий"`, `"hook для видео"`, `"структура ролика"`, `"скрипт ролика"`
</details>

<details>
<summary><b>🎞 yt-montage</b> — монтажный лист</summary>

Превращает готовый сценарий в монтажный лист: что показывать на экране в каждый момент, b-roll / shot list. Faceless-визуал (без говорящей головы). Текст не пишет — потребляет готовый сценарий.

**Триггеры:** `"монтажный лист"`, `"распиши монтаж"`, `"что показывать в видео"`, `"раскадровка"`, `"b-roll лист"`, `"план монтажа"`
</details>

<details>
<summary><b>🔎 yt-seo</b> — название, описание, теги</summary>

Заголовок (A/B варианты), описание с таймкодами, теги.

**Триггеры:** `"название и описание"`, `"тайтл и теги"`, `"подбери название"`, `"опиши видео"`, `"теги для видео"`

**Что НЕ делает:** превью — это `yt-thumbnail`.
</details>

<details>
<summary><b>🖼 yt-thumbnail</b> — концепт превью</summary>

Идея превью: текст, композиция, цвет, эмоция. **Картинку не рисует** — это концепт, который ты потом нарисуешь сам или в инструменте.

**Триггеры:** `"превью для видео"`, `"обложка для видео"`, `"thumbnail концепт"`, `"что нарисовать на превью"`
</details>

<details>
<summary><b>🎯 yt-promo</b> — продвижение и лидген на канале</summary>

**On-channel only:** pinned-комменты, end-screens, CTA в видео и описании, воронка на сайт-визитку.
**Out of scope:** кросс-промо на других платформах (Telegram, Habr и т.п.).

**Триггеры:** `"продвижение канала"`, `"как раскрутить канал"`, `"лидген на канале"`, `"привлечь заказы"`, `"закреп под видео"`, `"end screen для видео"`
</details>

### Agents

|-------|-----------------|------------|
| ♟ **yt-strategist** | `"что снимать на этой неделе"`, `"топ идеи на неделю"`, `"стратег по каналу"`, `"yt-strategist"` | Читает базу `yt-knowledge-base` + свежие отчёты конкурентов из vault, синтезирует **топ-5 приоритетных идей** на неделю с обоснованием на данных. ## 🛠 Поддержка и развитие

### Где править

| Что | Чем |
|-----|-----|
| Новый плагин | `/create-plugin` (из `plugin-builder`) |
| Новый скилл/агент в существующем плагине | `/extend-plugin` (из `plugin-builder`) |
| Точечная правка скилла/агента после инцидента | `/improve-plugin` (из `plugin-builder`) |

> ⚠️ **Не редактируй** руками файлы в `~/.claude/plugins/cache/` — это read-only кэш, изменения затрутся. Реальный источник — этот репозиторий, путь зависит от машины. На текущей машине: `~/.claude/plugins/marketplaces/anton-toolkit-marketplace/` (он же — git working copy).

### Версионирование

Каждая правка плагина → **bump PATCH** в `plugins/<name>/.claude-plugin/plugin.json`. Без bump'а маркетплейс не подтянет изменение на других машинах.
